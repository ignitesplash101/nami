## Gotchas

- **Scenario cache key MUST include all semantic inputs.** `scenario_cache_key()` hashes scenario_text + portfolio_key + holdings dict + market_date + model_id + prompt_version + factor_universe_version + events_version + **regression_spec** (required â€” the engine-math lever, see below), **plus position_quantities and pinned_event_ids when present**. Dropping any of these would silently return wrong P&L (e.g. same tickers + different weights produces different P&L but would have hit the same cache entry).
- **`regression_spec` is the engine-math cache lever (Phase 18); PROMPT_VERSION is the prompt/schema lever.** `app/factors/regression.py::regression_spec()` renders `"<REGRESSION_ESTIMATOR_ID>|lookback=â€¦|alpha=â€¦|min_obs=â€¦"` and is a REQUIRED kwarg of `scenario_cache_key` â€” changing the estimator math (bump `REGRESSION_ESTIMATOR_ID`), `RIDGE_ALPHA`, or `BETA_LOOKBACK_WEEKS` self-invalidates every cached result. Do NOT bump PROMPT_VERSION for engine-math changes.
- **Gemini grounding + `response_schema` in one call is unreliable.** Gemini Flash can return valid JSON while skipping `google_search`, leaving `grounding_metadata` empty (observed on `gemini-3.5-flash`; the split deliberately stays on `gemini-3.6-flash`). `propose_shocks_with_retry` deliberately splits Call 2 into `_grounded_narrative` (text + Google Search, no schema) and `_extract_structured_shocks` (schema, no tools). Do NOT merge them back together or add a "retry without grounding" fallback.
- **Grounded narratives use a preference hierarchy, not a brittle allowlist.** Prompt v11 asks Google Search grounding to prefer government, central-bank, regulator, and exchange sources; then peer-reviewed papers, working papers, and institutional research; then major financial/general news. Broker marketing, retail-finance/crypto sites, SEO explainers, and Wikipedia are discouraged whenever stronger evidence exists. Keep the narrative at 3â€“5 sentences and preserve the grounding/schema split.
- **`run_scenario` accepts injected `gemini` and `cache`** â€” tests use mocks (`tests/conftest.py::InMemoryCache` and `_MockGeminiClient`) instead of `storage.Client()` and `genai.Client()`. Production code calls with defaults (which read `Config` and construct real clients).
- **PROMPT_VERSION** in `app/llm/prompts.py` MUST be bumped for ANY change that affects `ScenarioResult`'s shape OR prompt semantics. Schema changes invalidate the cache exactly the same way prompt changes do. v3 means `ScenarioResult` carries `portfolio_name` + `portfolio_holdings`.
- **`ScenarioResult` is self-contained as of v3.** Always render holdings from `result.portfolio_holdings`, never from `get_portfolio(result.portfolio_key)` â€” the latter raises on `"custom"`. The new fields have defaults so cached v2 entries don't crash, but they'll show weight=0 for tickers; the PROMPT_VERSION bump invalidates them lazily.
- **`run_scenario` accepts EITHER `portfolio` (positional, `str | Portfolio`) OR `portfolio_key=` (kwarg, back-compat).** Passing both or neither raises `ValueError`. The UI passes a `Portfolio` object so custom holdings round-trip; existing tests still pass `portfolio_key="us_tech_growth"`.
- **`tests/test_live_evals.py` is network-gated on `RUN_NETWORK_TESTS=1`** and **stochastic over news drift** even with `temperature=0` â€” use semantic assertions only (tag membership, ordering, presence-of-citation), never magnitude bounds. `docs/backtest_results.md` is a DATED snapshot, not a permanent benchmark. Costs ~$0.08 per test (gemini-3.6-flash list prices incl. thinking tokens, measured 2026-07-22).
- **Theme sensitivity is experimental fixed-context narrative Shapley**, not causal risk attribution. Each subset PINS the source scenario's analog set and uses the analog-only narrative path (no re-selection, no re-grounding) â€” only the shock proposal varies per fragment, so subset payoffs are deterministic. Frame this honestly in both UI (section label says "Experimental") and methodology doc.
- **Narrative Shapley lives in `app/llm/narrative_shapley.py::compute_narrative_shapley`, NOT inside `run_scenario`.** The UI calls `run_scenario` first, then `compute_narrative_shapley` if opted-in, which calls `decompose_scenario` + reruns the pipeline 2^N âˆ’ 1 times and attaches the result via `model_copy(update={"narrative_shapley": ...})`. Putting it inside `run_scenario` would invite recursion and control-flow confusion.
- **`PROMPT_VERSION` covers cache invalidation for prompt OR schema changes.** v3 â†’ v4 because `PortfolioPnL.by_factor` was renamed to `by_factor_naive` + new field added + `ScenarioResult.narrative_shapley` introduced. v4 â†’ v5 added `by_factor_conditional_shapley_explicit` and `by_factor_conditional_shapley_grouped`. The anticipated PROMPT/engine split happened in Phase 18: engine math is now keyed by the separate `regression_spec` cache-key component, so PROMPT_VERSION stays a pure prompt/schema-semantics marker.
- **N must be in [2, 4] for narrative Shapley.** Both `decompose_scenario` and `compute_narrative_shapley` raise `RuntimeError` if the decomposer returns out-of-range. N=5 would be 31 subset runs â‰ˆ 8 min sync wall-clock and is cut off intentionally.
- **Narrative Shapley runs subsets in parallel.** `compute_narrative_shapley` submits all 2^Nâˆ’1 non-empty subsets to a `ThreadPoolExecutor` (`config.narrative_shapley_max_workers`, default 4) and drains via `concurrent.futures.as_completed` so the progress callback fires from the main thread. `subset_pnls` is keyed by mask, not by completion order â€” the Shapley math is deterministic regardless of how the pool drained. If Gemini rate-limits show up in practice, drop `NARRATIVE_SHAPLEY_MAX_WORKERS=2`.
- **`run_scenario` parallelizes portfolio-prices vs factor-history fetches.** Inside the cache-miss path, `fetch_weekly_prices(portfolio.tickers)` and `get_factor_returns_with_history(...)` are submitted to a 2-worker ThreadPoolExecutor and joined before `estimate_betas_for_portfolio` runs. The combined factor fetch returns `(raw_returns, demeaned_history_or_None)` â€” passes BOTH into the beta estimator AND the SHAP background, eliminating a duplicate yfinance round-trip that existed pre-Phase-9.
- **Mocks for `run_scenario` MUST stub FOUR market-layer functions** in the scenario module's namespace (per the parallelized hot path): `app.llm.scenario.fetch_weekly_prices`, `app.llm.scenario.get_factor_returns_with_history`, `app.llm.scenario.estimate_betas_for_portfolio`, and `app.llm.scenario.convert_weekly_returns_to_usd` (identity stand-in â€” keeps non-USD books network-free). See `tests/test_scenario.py::_patch_market_layer`. The beta-estimator mock must accept `**kwargs` AND return the `(betas, stats)` tuple (`stats` is `dict[str, TickerRegressionStats]`) because the orchestrator unpacks it.
- **yfinance is hoisted above the Gemini chain in `run_scenario`** (Phase 10). The ThreadPoolExecutor that fetches portfolio prices âˆ¥ factor history is opened immediately after the cache check and stays open across `select_analogs` â†’ `compute_envelope` â†’ `propose_shocks_with_retry`. yfinance becomes effectively free (~2-6s) by overlapping with the ~8-16s Gemini chain. Don't move it back below the Gemini calls "for clarity" â€” it's a deliberate parallelism win.
- **Iterative shock adjustment is structured editing, NOT free-form re-proposal.** `adjust_scenario_shocks` in `app/llm/scenario.py` takes a `cache_key` (NOT a client-supplied `prior_result`) and either `overrides: dict[str, float]` (manual sliders) or `adjustment_text: str` (LLM prompt). The server re-fetches the trusted canonical from cache. Narrative, citations, analogs_selected, periphery_shocks, and factor_envelope are guaranteed byte-for-byte unchanged by `model_copy`, not by prompt wording. Only `factor_shocks`, `portfolio_pnl`, and `adjustment_history` change. Derived results are NOT cached.
- **The LLM patch is a separate schema (`ShockEditPatch`), NOT `ShockProposalOutput`.** Reusing the proposal schema would let the model rewrite narrative/periphery, which violates the editing invariants. `ShockEditPatch` carries `scope: Literal["local", "rerun_required"]` + `edits: list[FactorEdit]` + `rejection_reason: str | None`. `scope="rerun_required"` surfaces via HTTP 422 with `rejection_reason` as the detail; the UI offers a "Rerun full scenario" CTA pre-filling the original text + adjustment. Semantic changes (new mechanism, new region, new factor) belong in a rerun, not an edit.
- **Zero-removal carve-out in `app/llm/adjust_validation.py::validate_factor_overrides`:** `new_shock == 0.0` is ALWAYS accepted regardless of envelope (explicit removal sentinel). Otherwise must be in `[p10, p90]` **when the factor's analog `count >= 3`**; for low-evidence factors (`count < 3`, or missing envelope/p10/p90 â€” incl. the count=0 rows stored as 0/0/0) the only valid values are 0.0 or the canonical shock itself (**keep-or-remove**, `math.isclose` echo check). Note `validate_shock_proposal` is NOT advisory either â€” its band check (countâ‰¥3) blocks after a one-retry repair loop; adjustment just rejects immediately and adds the keep-or-remove rule. The validator also requires `set(overrides.keys()) == set(canonical_factor_names)`; missing or extra keys raise. UI mirrors this by rendering one slider per canonical factor and disabling re-tuning on `count < 3` rows ("n=K analog observation(s) â€” keep or remove"); the rule binds LLM patches and direct API calls server-side regardless.
- **`ScenarioResult.adjustment_history: list[ShockAdjustment]` is a derived-only field with `default_factory=list`.** Canonical results round-tripped through the GCS cache always carry `[]`. Adjusted results carry a growing list, but adjusted results are never cached. Because the field has a default, the schema change did NOT require a `PROMPT_VERSION` bump â€” old v5 cached entries deserialize with `adjustment_history=[]`.
- **Backdating contract (Phase 11):** `run_scenario(market_date=...)` is the as-of date. When `effective_as_of < today` (resolved via `app/utils/calendar.py::resolve_effective_market_date` â€” NYSE trading day on or before the requested date), the pipeline switches to vintage-controlled backdated mode. Three things change at once: (1) the events registry is filtered by `event.end_date <= effective_as_of` via `app/factors/analogs.py::filter_events_as_of` (end-date, **not** start-date â€” start-date filter would let in-progress events leak post-as-of returns through `fetch_event_returns`); (2) yfinance fetches use `end=effective_as_of + 1 day` (yfinance `end=` is exclusive); (3) the warm cache (`get_factor_returns_with_history`) is BYPASSED â€” call `fetch_factor_returns_with_history(end=...)` directly â€” because the warm cache keys only on `lookback_weeks` and always returns current data; (4) the narrative call goes through `_analog_grounded_narrative` (no Google Search) and returns empty citations.
- **`adjust_scenario_shocks` is also vintage-controlled.** When the canonical scenario's `market_date < today`, the adjustment refetches with the same `end=canonical.market_date + 1d` and also bypasses the warm cache. Without this, the new P&L would mix backdated shocks with current betas â€” a look-ahead leak.
- **Effective vs requested as-of dates.** `result.market_date` is the **effective** date (the NYSE trading day used for filtering and data fetches). `result.requested_as_of_date` (new Phase-11 field) is the raw user request. The UI displays both in the backdated-mode banner. The cache key uses `effective_as_of` so a user picking a Saturday hits the same cache as someone picking the prior Friday.
- **`PROMPT_VERSION` is at v11 in code.** v11 added the grounded-narrative source hierarchy above. v10 added the four Phase-22 factors (EFA/HYG/GLD/SHY) to the LLM payload and EFA to rule 7's overlap enumeration; the registry grew 17 -> 31 alongside (that part rides `events_version`). The v9 shock-semantics contract still holds. Treat `app/llm/prompts.py` as authoritative.
- **LLM parametric knowledge is NOT vintage-controlled** even on the backdated path. Disabling Google Search prevents current-news leakage but the Gemini model still "knows" about events after the as-of date (e.g. it knows COVID happened even if as_of is 2018). The UI banner (`frontend/src/AsOfDatePicker.tsx::BackdatedModeBanner`) makes this honest. For sign-off-grade reproducibility, treat the analog envelope and shock magnitudes as canonical; the narrative is illustrative.
- **`ScenarioResult` schema gained four new fields in Phase 11**: `requested_as_of_date` (defaults None), `narrative_mode` ("grounded" | "analog_only", defaults "grounded"), `selected_event_ids` (defaults `[]`), and they're all default-defaulted so v5-cached payloads deserialize cleanly under v6 lazy re-derivation.
- **`MIN_ELIGIBLE_ANALOG_EVENTS = 2`** in `app/llm/scenario.py`. Backdating to a date that leaves fewer eligible analogs raises `ValueError` â†’ 422. The pre-2010 portion of `historical_events.yaml` is sparse so very-old as-of dates surface this gracefully rather than failing on a Gemini call.
- **Mock `_MockGeminiClient.propose_shocks_with_retry` accepts `analog_grounded`, `as_of_date`, `selected_analog_events`, and `per_event_returns`.** Tests that exercise backdated paths use `type(gemini).last_analog_grounded` to assert which narrative path was taken.
- **Mock `_patch_market_layer` captures `end=` kwarg** so backdating tests can assert that yfinance was called with the correct effective-as-of + 1 day. The captured dict is the return value of `_patch_market_layer(monkeypatch)`.
- **`run_scenario` validates `selected_ids âŠ† events` and raises `ValueError` on a hallucinated analog id** (â†’ 422). Without it, `compute_envelope` raises a bare `KeyError` that the `/run` endpoint (which only catches `ValueError`) would surface as a 500. The SSE path already catches `Exception`.
- **`decompose_scenario` validates the cached payload** (`sub_narratives` must be a non-empty `list[str]`) before trusting it; a corrupted/old-format entry recomputes instead of crashing downstream.
- **NOT a bug â€” do not "fix": weekend/holiday live runs are not mis-flagged as backdated.** `resolve_effective_market_date` returns `requested` unchanged when `requested >= today`, so `is_backdated = effective_as_of < today` is equivalent to `requested < today`. A Saturday live run keeps the live Google-Search path.
- **Phase 12 needed NO `PROMPT_VERSION` bump.** The visual redesign, a11y wiring, and correctness hardening change neither `ScenarioResult`'s shape nor prompt semantics, so cached entries stay valid.
- **`run_scenario` quantity mode marks BEFORE the Gemini chain** (`_apply_mtm` / `mark_book` in `app/llm/scenario.py`): the price-derived weights flow into the prompt AND the engine. The GCS scenario cache stores the **return-space canonical + the quantity INPUTS only**; NAV / marks / FX are attached **post-retrieval** (on hit and miss) and are **never cached** â€” so the same scenario at a different NAV can never serve stale dollars (finding #1 from the design review). The cache key folds `position_quantities` only when present, so weight-only keys are byte-identical to pre-MTM. The NAV-scalar `portfolio_nav` is never keyed and never cached (pure overlay).
- **`adjust_scenario_shocks` re-marks** from the canonical's cached `position_quantities` at `canonical.market_date` (vintage-correct) and re-attaches NAV; dollars stay client-side. MTM inputs (quantities/NAV) are **admin-only** (`_resolve_mtm` in `app/api/main.py`). Reporting currency is **USD only** in v1.
- **Phase 13 (MTM) needed NO `PROMPT_VERSION` bump** â€” marking is post-LLM and all new `ScenarioResult` fields are optional with defaults, so old cached payloads deserialize as return-only.
- **`run_scenario(pinned_event_ids=â€¦)`** (theme-sensitivity subsets): skips `select_analogs`, validates `pinned âŠ† events`, and forces the analog-only (no-Search) narrative via `use_analog_only = is_backdated or (pinned_event_ids is not None)`. **`narrative_mode` keys off `use_analog_only`, NOT `is_backdated`.** Folded into the cache key only when present (non-pinned keys byte-identical).
- **Theme sensitivity is fixed-context narrative Shapley, NOT "more realistic Shapley".** `compute_narrative_shapley` pins the source scenario's analogs across all subsets + skips re-grounding â†’ deterministic `v(S)`. It measures the marginal shock each theme adds *within the original analog context* (illustrative, not causal). Served over SSE **`/api/scenarios/decompose-stream`** (mirrors `run-stream`; emits `{stage:"subset", done, total}` so the UI shows "X / Y subset runs"). Dollarized in the UI as `shapley_value Ã— NAV` when a NAV exists.
- **Phase 14 needed NO `PROMPT_VERSION` bump** â€” the notional NAV is post-pipeline client display; `pinned_event_ids` is inside-pipeline with an optional input; `ScenarioResult` shape is unchanged.
- **Prompt/schema changes are currently v11.** If a later change affects prompt semantics or `ScenarioResult` shape, bump `PROMPT_VERSION` again and update README, `docs/llm-system-design.md`, `docs/methodology.md`, `CLAUDE.md`, and `AGENTS.md` together. Changing factor `description` strings in `app/factors/universe.py` ALSO requires a PROMPT_VERSION bump (descriptions feed the LLM payload but `factor_universe_version` deliberately ignores them). Engine-math changes do NOT bump it â€” they ride `regression_spec`.
- **`Portfolio` gained a `benchmark: str | None` field; benchmark P&L is a NON-cached post-cache overlay** (mirrors MTM). `_benchmark_overlay` runs the benchmark ticker as a one-holding portfolio through the result's own shocks and computes total P&L plus naive attribution only; the three conditional attribution fields remain `null`. It then attaches `benchmark_ticker` / `benchmark_pnl` / `active_return = total âˆ’ benchmark_total` via `model_copy`. It is best-effort and never cached, so the canonical blob never carries the overlay.
- **`adjust_scenario_shocks` now rebuilds the portfolio from `canonical.portfolio_holdings` for ALL portfolios** (dropped the `== "custom"` special-case). Rebuilding *sample* books via `get_portfolio(key)` would silently recompute against the *current* snapshot weights, so after a weight refresh an adjustment of an old cached scenario would drift off its canonical. Because benchmarks are overlay-only, `adjust` can't recover a custom book's benchmark from the canonical â€” `ScenarioAdjustRequest.benchmark` is **resent by the client** (`AdjustmentPanel` passes `result.benchmark_ticker`); sample books fall back to `sample_benchmark(portfolio_key)`.
- **`CASH` is the zero-exposure sentinel** (`CASH_TICKER` in `sample_portfolios.py`). It is stripped before the yfinance/beta fetch and before LLM periphery selection; `_estimate_betas_cash_aware` runs the estimator on a cash-free (re-normalized â€” betas are weight-independent) book then injects an all-zero `CASH` beta row, so `portfolio_pnl` carries CASH at exactly 0 contribution while its weight dilutes the rest. In MTM mode a **`CASH` quantity is a USD amount, not shares** (`mark_book` excludes it from `fetch_marks`, marks 1.0 â†’ value == quantity). `validate_holdings`/`validate_quantities` accept the sentinel but **reject an all-cash book** (no market exposure to shock). Do NOT add CASH handling to `app/factors/shocks.py` â€” it lives in `scenario.py`.

- **(Phase 18) Periphery shocks are hard-banded.** `validate_shock_proposal` rejects non-finite values and `|shock| > MAX_ABS_PERIPHERY_SHOCK = 0.75` (blocking, retry-then-fail like every validator string).
