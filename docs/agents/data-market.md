# nami — market data, caching & marking (agent notes)

Owns `app/data/*` (yfinance wrapper + GCS market cache, FX, marking/MTM,
sample portfolios) and `app/utils/calendar.py`. Change a contract here →
update this doc in the same commit.

## Gotchas

- **yfinance silently drops tickers it can't fetch.** Any function that maps tickers → returns must validate set membership and raise loudly. Documented at `app/data/market.py` (the dropna step).
- **Parquet round-trip normalizes datetime precision** (`datetime64[s]` → `datetime64[ms]`). Values are identical; only the index dtype changes. Use `assert_frame_equal(..., check_index_type=False)` in tests.
- **`fetch_weekly_prices(..., lookback_weeks=N)` anchors on `end` if provided, else `utcnow()`.** Both `fetch_weekly_prices` and `fetch_daily_prices` share an internal `_fetch_prices` helper — if you touch one, verify the other.
- **yfinance's `end` is EXCLUSIVE.** `start=2020-02-19, end=2020-03-23` returns bars through 2020-03-20, NOT 2020-03-23. `fetch_event_returns` in `app/factors/analogs.py` adds 1 day before calling yfinance to get inclusive-end semantics; callers wanting "include the end date's close" must do the same +1-day transform.
- **yfinance `interval="1wk"` returns Monday-anchored bars, not week-ending Fridays.** `fetch_weekly_prices(start="2020-02-19", end="2020-03-23")` returns a first row dated **2020-02-17** (Monday). This breaks "first Friday ≥ start" intuition — Phase 3 uses `fetch_daily_prices` instead to get exact-day alignment for event windows.
- **yfinance is cached at the `_fetch_prices` layer.** `app/data/market.py` consults a process-wide `CloudStorageCache` (prefix `market_data`, parquet, 24h TTL) keyed on `(sorted_tickers, interval, start, end)` before calling `yf.download`. Cache failures (no GCS creds, parquet schema mismatch) silently fall back to yfinance — both read and write paths are wrapped in `contextlib.suppress(Exception)`. The singleton is `app/data/market_cache.py::get_market_cache()` (lru_cache(maxsize=1)); tests pass `cache=None` explicitly to bypass.
- **NYSE holiday list is hand-curated in `app/utils/calendar.py`** for 2020-2030. Pre-2020 / post-2030 dates fall through weekend-only logic (acceptable for v1). To extend, add dates to `NYSE_HOLIDAYS` or swap in `pandas_market_calendars`.
- **yfinance timeout lives at `app/data/market.py::_fetch_prices`** (`timeout=_YF_DOWNLOAD_TIMEOUT_SECONDS`, 30s). This is the ONLY correct place to bound the fetch: `run_scenario`, `adjust_scenario_shocks`, and `compute_envelope` wrap it in `with ThreadPoolExecutor(...)`, whose `__exit__` calls `shutdown(wait=True)` and re-blocks on a hung worker — so a `future.result(timeout=)` would NOT bound the request. One timeout here covers all three (every fetch routes through `_fetch_prices`).
- **Mark-to-market (Phase 13) is a LINEAR overlay, computed CLIENT-SIDE.** The engine is linear in weights, so every dollar figure is `return_field × portfolio_nav` (`PortfolioPnL` has NO dollar fields by design — the frontend multiplies). Per-position `$ P&L = by_ticker_total[t] × NAV` (the weight cancels — no unweighted-return field needed). Don't add dollar P&L to `PortfolioPnL`/`portfolio_pnl`.
- **Marking lives in `app/data/marking.py`.** `currency_for_ticker` (Yahoo-suffix map; `.L` → `GBp`, price ÷100), `fetch_marks` (dedicated **daily RAW close**, `auto_adjust=False`, bounded window, last-valid-close ≤ as_of per instrument + staleness guard), `fetch_fx_to_usd` (explicit pair+inversion map — e.g. JPY via `USDJPY=X` **inverted**), `mark_positions`, `mark_book`. All data-unavailable failures raise `MarkingError` (a `RuntimeError` subclass) → API maps to **503**.
- **MTM marks MUST use raw close, not the adjusted weekly `ticker_prices`.** Valuing a share count needs the actual traded price (`auto_adjust=False`); `market_cache_key` carries an `auto_adjust` flag so raw and adjusted never collide. The flag is appended only for raw fetches, so existing adjusted-close cache keys are unchanged.
- **MTM is fail-closed.** If a requested valuation can't be marked (missing/stale price, unavailable FX) the run raises → HTTP **503**, never a percentage-only result. Best-effort degrade is only acceptable for non-requested display adornments (n/a here).
- **Sample portfolios load cap-weights from a frozen snapshot, NOT hard-coded dicts.** `app/data/sample_portfolios.py` reads `sample_portfolio_weights.json` at import (eager `SAMPLE_PORTFOLIOS = _build_portfolios()`); a missing/incomplete snapshot raises at import (hard-fail by design — never silently equal-weight). Regenerate with `uv run python scripts/refresh_sample_weights.py`. The script caps-weights via yfinance `.info["marketCap"]` per portfolio (single-currency assertion: `msci_world` is all-USD via ex-US ADRs, `japan_equity` all-JPY) and bakes sector/country tags in the same pass. Issuer CSVs (iShares/Invesco) are the *preferred* weight source but their endpoints serve an HTML SPA to headless clients; the JSON's `sources` field records provenance so a networked re-run can upgrade it. Re-weighting changes each book's `holdings` → new `scenario_cache_key` (natural miss), so NO `PROMPT_VERSION` bump.
- **(Phase 18) Non-USD ticker returns are USD-converted BEFORE beta estimation.** `app/data/fx.py::convert_weekly_returns_to_usd` compounds `(1+r_local)(1+r_fx)−1` using `marking.py`'s suffix/pair maps (new public accessors `major_currency` + `fx_pair_for_currency`); wired after `compute_weekly_returns` in BOTH `run_scenario` and `adjust_scenario_shocks` with the vintage-correct `end=`. All-USD books pass through identity with ZERO FX fetches; an entirely-missing FX series raises `MarkingError` → 503 (fail-closed — silently regressing local-currency returns would reintroduce the JPY-portfolio-vs-USD-benchmark active-return mismatch this fixes). GBp converts with GBPUSD directly (the /100 cancels in pct_change).
- **(Phase 35) Complete historical-event returns persist for 30 days in GCS parquet.** `app/factors/warm_cache.py` uses the `event_returns` prefix and keys the matrix by the event-registry version, factor-universe version, market-cache version, and cache-format version. The loader accepts partial NaNs for pre-launch factors but rejects structural axis mismatches and any entirely missing event row. A scenario never populates all 31 events on a cache miss: it fetches only the selected analog rows; the replay surface owns full population and background warming.
- **(Phase 19) The market cache stores COMPLETE batches only.** A ticker yfinance flakes on (rate limit/outage) comes back as an all-NaN COLUMN that survives the row-wise dropna; `_fetch_prices` now drops all-NaN columns (making the documented "failed tickers are dropped" contract true) and skips the cache write when `set(columns) != set(requested)` — caching an incomplete batch would poison every run on that window for the 24h TTL (the real-world symptom was `InsufficientHistoryError: NVDA (n=0)`). The n=0 error message now names the transient-fetch cause and says retry; `/run` maps remaining bare RuntimeErrors ("yfinance returned no data…") to a coded 503 `unavailable` (clause must stay LAST — MarkingError/InsufficientHistoryError subclasses take their own statuses).
