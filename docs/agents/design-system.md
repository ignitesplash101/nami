# nami — design system & responsive CSS (agent notes)

Owns `frontend/src/styles.css` + `theme.ts` + `copy/` (tokens, dual theme,
fluid scale, motion, breakpoints, InfoTip/glossary). Change a contract here
→ update this doc in the same commit.

## Gotchas

- **Responsive breakpoints (Phase 10c):** `@media (max-width: 1079.98px)` is the tablet/phone-landscape tier; `@media (max-width: 640px)` is the phone tier; baseline (≥1080px) is desktop. The desktop boundary is 1080px (NOT 1024px) because `.app-shell` requires 360px rail + 720px workbench = 1080px to render without horizontal overflow. iPad Pro portrait at 1024×1366 renders in tablet mode by design.
- **`body { min-width: 1100px }` is gone (Phase 10c).** Do not reintroduce — it forces horizontal scroll on every viewport below 1100px. The responsive grid below handles layout.
- **Mobile-only CSS rules live at the bottom of `frontend/src/styles.css`** in two `@media` blocks. Above the blocks is the desktop baseline; the blocks layer overrides. `body { min-width }` was REMOVED, not gated. The rail's `height: 100vh` is paired with a follow-up `height: 100dvh` declaration so iOS Safari's collapsing URL bar doesn't leave a gap.
- **Tables: the mobile `.table-scroll > table { min-width: 480px }` rule is scoped — NOT a blanket `table` rule.** Three table surfaces exist: (1) tables inside `.table-scroll` (factor shocks, name-level, analogs, theme sensitivity) — the mobile rule applies; (2) markdown-rendered tables inside `.section-content` (methodology drawer) — they get `overflow-x: auto` on the section instead; (3) `.adjustment-history` list — narrow enough to leave alone. If you add a new wide table, wrap it in `<div className="table-scroll">` so it gets the responsive treatment.
- **Hamburger button class `.rail-toggle-btn` extends `.methodology-btn`** styling for visual consistency. It's `display: none` baseline and switches to `display: inline-flex; order: -1` at `@media (max-width: 1079.98px)` (the `order: -1` floats it to the left of the topbar's other children).
- **Touch-target baseline is now 40px** on `.primary-button` / `.ghost-button` / `.segmented button` (was 38px), bumping to 44px at ≤1079.98px. Inputs got `padding: 12px` (was `10px 11px`) baseline + `min-height: 44px` at ≤1079.98px. Slider thumbs are sized to 22×22 globally via `::-webkit-slider-thumb` / `::-moz-range-thumb` — applies to both desktop and mobile for parity.
- **iOS Safari `100dvh`**: `.app-shell` and `.rail` use `min-height: 100vh` followed by `min-height: 100dvh` (declaration-order fallback) so older browsers degrade to `100vh`. The `.drawer-panel` and `.rail-drawer-panel` use `top: 0; bottom: 0` fixed-inset positioning, so no 100vh involved there. `.rail-drawer-panel` and `.drawer-body` add `overscroll-behavior: contain` to prevent rubber-band scroll propagating to the body.
- **Phone-stack pattern for label+icon+input rows.** When a `<label>` contains an icon, text, AND an input (e.g. `.asof-picker label`), the inner content is inline-flex on desktop. On phone you want the icon+text inline on row 1 and the input full-width on row 2 — a flex direction:column on the outer container is NOT enough. Use a 2-column grid on the label with `grid-column: 1 / -1` on the input. See the `@media (max-width: 640px)` block at the bottom of `frontend/src/styles.css`.
- **`.saved-row-head` and `.saved-row-meta` both stack on ≤640px.** Both use `justify-content: space-between` between a left-aligned cell and a right-aligned counterpart; on narrow phones long names would crush the right cell. The phone media-query rule stacks them; the `.saved-open` tap target is bumped to `min-height: 40px` at the same breakpoint since the button is now visually isolated.
- **Phase 12 design system ("Hokusai Deep").** `frontend/src/styles.css` `:root` holds the full palette (`--bg` Prussian indigo, `--accent` sand, `--accent-2` crest-blue, `--up`/`--down`, `--warn`) plus type-scale / radius / shadow / motion / `--focus` / font tokens. The legacy names (`--panel`, `--panel-2`, `--red`, `--green`, `--cyan`, `--amber`) are kept as **aliases** of the new tokens, so every existing `var(--*)` reference re-skinned for free. The alias trick does NOT touch hard-coded literals — new CSS must use tokens, not raw hex/rgba, or it will read as the old theme. CTA (`.primary-button`) is sand-on-ink, NOT red (red is reserved for loss/error).
- **`--cyan`/`--amber` are aliases now, not literal colors.** `--cyan → --accent-2` (crest blue), `--amber → --warn`. "Active" affordances (segmented/nav/tag) are crest-blue on ink. Don't reintroduce literal `#8bd3ff`/`#f6b84b`.
- **The brand mark is the 波 ghost glyph + seigaiha crest** (`App.tsx` `.brand-block`). Both are decorative: `aria-hidden` + `pointer-events: none`. The crest is a CSS `background-image` data-URI wave, not an `<img>`. Keep `.brand-glyph` anchored fully INSIDE the block (small positive `top`/`right` insets + a size that fits the ~280–312px rail) — `.rail`'s `overflow: auto` and `.brand-block`'s `overflow: hidden` clip any off-edge bleed, so negative offsets visibly cut the kanji.
- **SSE run-progress is the "wave propagating" signature — per-item, NOT a continuous track.** Each `.run-progress li.active .dot` emits an expanding `::after` ripple (`@keyframes nami-ripple`); done stages settle to a filled `--up` node. The `<ol>` wraps to rows (and goes vertical ≤640px), so do NOT add a single L→R track spanning it — it would cut across wrapped rows.
- **Results "ripple" reveal fires once per mount, not per re-render.** `.results-stack > *` runs `nami-rise` with a per-child stagger. React keeps these nodes across attribution-toggle re-renders, so the animation only plays when a run first lands. Don't key the stack on `attributionMethod` or it will re-animate on every toggle.
- **`prefers-reduced-motion` is handled by one global block** at the bottom of `styles.css` (neutralizes every animation/transition + `scroll-behavior`). `MethodologyDrawer.tsx` ALSO gates its `scrollIntoView` via `scrollBehavior()` (JS can't read the CSS block). New JS-driven smooth scrolls must call `scrollBehavior()` too.
- **Keyboard focus = global `:where(...):focus-visible` ring** (`--focus` double-ring). Inputs keep `outline: none` + a `:focus` border-color; the ring is the keyboard affordance layered on top. Don't re-add bare `outline: none` to buttons without a `:focus-visible` replacement.
- **(Phase 19) `.results-toolbar` is sticky ≥1080px under `--z-sticky` (10)** — the three `max-width` breakpoints are untouched (there are now TWO `min-width` queries: 1080 sticky-toolbar, 1440 wide grid). New sticky elements slot below `--z-backdrop`. The stale-results treatment is a class toggle (`.results-stack.is-stale`) on the SAME nodes so the one-shot `nami-rise` stagger does not replay; `filter: saturate()` creates a stacking context, which is safe because the sticky toolbar and `th` live inside it.
- **(Phase 19) Wide screens are WIDE, not capped-narrow:** `--workbench-max` is 1800px (fills a 1920px screen completely; only ultrawides center with gutters), and at `@media (min-width: 1440px)` `.results-stack` becomes a `minmax(0,3fr) minmax(0,2fr)` grid where `.waterfall-card` + `.exposure-card` share a row (everything else spans `1 / -1`). The exposure card renders AFTER the waterfall in DOM (order = visual order, stagger unaffected); a `:not(:has(> .exposure-card))` fallback lets the waterfall reclaim the full row when the breakdown is empty. Prose stays readable via `.narrative p { max-width: 75ch }`, NOT by capping the layout.
- **(Phase 23) Mobile/zoom conventions from the QA hardening:** (1) grid tracks that can contain tables or long mono strings must be `minmax(0, 1fr)` — NEVER bare `1fr` or an implicit template; a track's `auto` minimum adopts the widest child's min-content and the ≤640px `.table-scroll > table { min-width: 480px }` rule then forces the workbench wider than a phone viewport (`.results-stack` base, `.app-shell`/`.scenario-grid`/`.two-column` compact overrides all carry the guard now, matching the ≥1440 grid). (2) CSS font sizes are rem-only — px font sizes don't respond to browser text-size settings; the 20 legacy px declarations were converted (10px=0.625rem … 14px=0.875rem). (3) The ±1σ idio-band "dispersion floor — not a confidence interval" framing renders as a visible caption under `@media (hover: none)` (`.idio-band-note`) because `title=` tooltips are unreachable on touch. (4) Touch-target additions at ≤1079.98px: `.chip`/`.guide-toggle`/`.drawer-close` 44px, `.chip-copy-btn` 40px (secondary inline affordance), and `.nav-knob input` re-asserted to 44px (its 34px baseline ties the generic input rule on specificity and wins on source order). (5) React dev-mode logs "Expected static flag was missing" at the ResultsPanel `<Plot>` — a known React 18 StrictMode × react-plotly.js internal false positive; dev-only, not actionable here.
- **(Phase 29) Fonts load from `index.html`** (two preconnects + `<link rel="stylesheet">`), NOT a CSS `@import` — the import serialized the font CSS behind the app stylesheet. Keep the family list in index.html in sync with the `--font-*` tokens in styles.css.
- **(Phase 31) Dual theme: `:root` is the DARK base ("Hokusai Deep"); `:root[data-theme="light"]` ("Hokusai Air" — Prussian ink on washi cream) overrides EVERY color-bearing token.** The inline boot script in `frontend/index.html` sets `data-theme` before first paint (explicit `nami-theme` localStorage choice wins, else `prefers-color-scheme`), so there is deliberately NO `prefers-color-scheme` branch in CSS — one source of truth. Never hard-code hex/rgba in a rule: a literal renders in ONE theme only. Add a purpose token in `:root` + a light override instead (the "Purpose tints" group is the pattern).
- **(Phase 31) `chartTheme()` is memoized PER THEME** — the cache key is `<html data-theme>`, so a theme toggle self-invalidates on the next read; `useTheme()` (`frontend/src/theme.ts`) re-renders the chart consumers. The hook owns the flip: sets `data-theme`, updates the `theme-color` meta, persists `nami-theme` to localStorage; OS scheme changes are followed ONLY until the user toggles explicitly. Keep the hex pairs in `theme.ts::THEME_COLORS`, the index.html boot script, and `--bg` (both themes) in styles.css in sync.
- **(Phase 31) `--accent-ink` is sand-as-FOREGROUND** (sortable-header text, the brand crest, the light-theme focus ring): `var(--accent)` in dark, darkened `#8a6a1c` in light. Sand text on cream fails contrast — never use `--accent` for text/icons; fills (CTA, slider thumbs) keep `--accent`.
- **(Phase 31) `.brand-crest` is a mask-image + `background-color: var(--accent-ink)`** — the data-URI stroke color is a throwaway black; converting back to `background-image` would bake one theme's sand into the SVG.
- **(Phase 31) The ≥1440px wide grid now lives INSIDE `.results-tabs .tab-panel`, not on `.results-stack`:** Drivers pairs waterfall (3fr) + factor table (2fr); Positions pairs exposure + name-level table; the `:not(:has(> .exposure-card))` fallback moved to the Positions panel's table. Don't re-add a `.results-stack`-level 2-track grid — waterfall and exposure are never visible together anymore.
- **(Phase 31) The `--step-*` type scale and `--space-5/6/7` are FLUID clamp()s** (360→1440px viewport interpolation; the 1.25 ratio holds at the wide end). Consumers keep using `var(--step-N)`/`var(--space-N)` unchanged — do NOT add per-breakpoint font-size overrides for anything already on the scale; extend the clamp instead.
- **(Phase 31) The rail track is `clamp(300px, 24vw, 380px)`** — it compresses toward 300px on small desktops (giving the 1080-1439 band a wider workbench) and caps at 380px on wide screens. 300 + 720 = 1020 < 1080, so the drawer-collapse boundary stays 1079.98px; don't shrink the 720px workbench minimum without re-deriving that boundary.
- **(Phase 31) The results tab-panel pairing starts at ≥1280px (workbench padding bump stays ≥1440px).** The empty-exposure fallback is `.tab-panel > .table-card:first-child { grid-column: 1 / -1 }` — scoped by POSITION, deliberately not `:not(:has(> .exposure-card))`, which would also match the Drivers panel (it never has an exposure card) and break its waterfall + factor-table pairing. Real bug caught in-browser at 1320px.
- **(Phase 31) Tablet band `@media (min-width: 720px) and (max-width: 1079.98px)` re-pairs `.two-column`** (2-up) between the drawer collapse and the desktop rail; it must stay AFTER the ≤1079.98 block in source order (same specificity — order decides).
- **(Phase 31) Copy layering: plain-first labels on the answer layer, quant names one level down.** `frontend/src/copy/glossary.ts` is the single source for plain↔quant term pairs; `copy/InfoTip.tsx` is the tap-friendly ⓘ popover replacing hover-only `title=` attrs (NOT a `useOverlay` — local listeners only, no scroll lock; one Esc may close an InfoTip and a drawer together, accepted). Relabels shipped: h1 "Portfolio scenario explorer", waterfall h3 "What drove the P&L" (+`.card-subtitle` keeping "Systematic contribution waterfall"), theme card "Which themes matter most", readout chip "± x single-name noise" (the old title text lives VERBATIM in `GLOSSARY.singleNameNoise.detail` — test-pinned), Evidence "N historical events · M sources", KnowYourBook "Understand this book — instant, free" with the Σ weight × beta formula moved into an InfoTip. NOT changed (deliberate): disclaimer, EvidenceBlock row labels + honesty captions, "Shock applied"/"P&L contrib", errorCopy, ComparisonPanel's "idio (1σ)" (pro surface), AdjustmentPanel's keep-or-remove caption.
