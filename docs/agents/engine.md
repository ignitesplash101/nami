# nami — factor engine & attribution (agent notes)

Owns `app/factors/*` (universe, regression, shocks, attribution, analogs,
warm_cache, engine_replay): the deterministic math, its contracts, and the
dated harness decision records. Change a contract here → update this doc
in the same commit.

## Gotchas

- **`portfolio_pnl()` returns JSON-safe dicts** (not pd.Series) since Phase 4 — Series would break the cache serialization round-trip. Shape includes `total_pnl`, the explicit attribution maps (`by_factor_naive`, conditional variants), and `by_ticker_factor` / `by_ticker_periphery` / `by_ticker_total`; all values are plain floats or dicts of floats.
- **`PortfolioPnL.by_factor_naive` is the persisted naive attribution.** There is NO `by_factor` alias. Pydantic v2's `computed_field` is included in `model_dump()` but rejected by `model_validate()` under `extra="forbid"`, which would poison the JSON cache round-trip. Readers must pick a specific variant: `by_factor_naive` / `by_factor_conditional_shapley` / `by_factor_conditional_shapley_explicit` / `by_factor_conditional_shapley_grouped`.
- **Four attribution maps live in `app/factors/attribution.py`, but only two are main UI choices.** `conditional_shapley_attribution_explicit` backs **Scenario shocks** (production risk view; explicitly shocked factors only), and `conditional_shapley_attribution_grouped` backs **Group totals** (risk-committee view). The grouped backend map remains factor-level after within-group redistribution; the UI waterfall sums it into true Market / Sector / Style / Macro bars and keeps factor detail in the table. `naive_attribution` and `conditional_shapley_attribution` stay in advanced diagnostics as **Naive algebra** and **Full conditional diagnostic**. The full diagnostic can allocate correlation credit to unshocked factors and must never drive the headline/top-driver readout.
- **The grouped Shapley implementation is "full Shapley + within-group sum-and-redistribute", NOT synthetic-feature aggregation.** Aggregating member coefs+shocks before running shap introduces within-group cross-product terms `c_f · s_{f'}` that don't exist in the linear model. Running full Shapley first then summing φ_f within each group preserves efficiency exactly.
- **`conditional_shapley_attribution_explicit` short-circuits the 1-feature case.** shap.LinearExplainer's covariance path crashes on a 1-D background; for a single explicit shock the trivial Shapley value `coef · shock` is returned directly.
- **Conditional Shapley ≠ causal attribution.** It is *data-dependent credit allocation* under the historical conditional distribution of factor returns. The **full** variant can attribute nonzero credit to factors with zero LLM shock (via correlation). In the UI those rows say `Correlation credit; no explicit shock`. The explicit/scenario view keeps unshocked factors exactly zero.
- **Factor labels are user-facing metadata, not model keys.** `app/factors/universe.py` exposes `key`, `ticker`, `group`, `short_label`, `display_name`, and `description`; `/api/factors` serves them read-only. Render as `Human label (TICKER)` wherever space allows, but do not change internal factor keys or math.
- **`ScenarioResult.risk_diagnostics` is warning-only deterministic post-processing.** It defaults to `[]` for old cached/saved payloads. Diagnostics may flag correlation conflicts, analog-envelope direction conflicts, and full-conditional cross-credit, but they must not rewrite shocks or clamp signs.
- **Use `shap.maskers.Impute(background)`** with the non-deprecated `LinearExplainer((coefs, intercept), masker)` form (see `app/factors/attribution.py`). The `feature_perturbation=` kwarg is deprecated in current shap; new code must avoid it.
- **Factor-return background must NOT `fillna(0)`.** `fetch_factor_returns_history` does `dropna(how="any")` then demeans, requiring ≥52 complete rows. Zero-filling a missing ETF manufactures false correlation that contaminates the Conditional Shapley values. The strict dropna restricts the background to the post-XLC-launch window (mid-2018+), which is correct.
- **Per-event analog fetching lives in `fetch_event_returns_matrix`** (Phase 18): events × factors matrix, input order preserved, up to 8 concurrent `fetch_event_returns` workers via `executor.map`. `compute_envelope` is now a thin wrapper (`compute_envelope_from_matrix(fetch_event_returns_matrix(...))`); `run_scenario` calls the two pieces directly so the per-event rows feed the extraction prompt + `ScenarioResult.analog_event_returns` without a duplicate fetch.
- **In-process warm cache lives in `app/factors/warm_cache.py`, NOT in `app/ui/`.** Since Phase 23 it is a lock-guarded dict that memoizes ONLY healthy results: a degraded `(raw, None)` tuple (SHAP background unavailable after a transient provider failure, e.g. a Yahoo rate limit at instance boot) is passed through but NEVER cached, so the next call retries instead of pinning every later scenario in the process to naive-only attribution (`tests/test_warm_cache.py` pins this). `app/factors/analogs.py::load_events`/`event_summaries`/`events_version` and `app/factors/universe.py::factor_universe_version` are still lru_cached (deterministic locals). Call `warm_cache.warm()` from a FastAPI `lifespan` hook to pre-populate.
- **The 3 Shapley fits in `portfolio_pnl` run in a 3-worker `ThreadPoolExecutor`** (Phase 10). `shap.LinearExplainer` drops into SciPy/NumPy linalg which releases the GIL, so threading buys real parallelism (~2-4s saved). Each variant still degrades independently to `None` on exception — the parallelism wraps the same `try/except` pattern, futures-style.
- **Scenario-shocks attribution sums to factor-driven P&L** under nami's demeaned-background contract — `fetch_factor_returns_history` demeans before passing to `shap.maskers.Impute`. Factors with shock=0 contribute zero to factor-driven P&L by definition, so the shocked-factor sub-game's grand-coalition value equals the factor-driven P&L. The distinguishing property is that **unshocked factors stay at exactly zero** (no correlation-driven credit). The regression guard is in [tests/test_methodology_worked_example.py](tests/test_methodology_worked_example.py).
- **Attribution invariants are pinned synthetically, not against doc numbers.** `tests/test_methodology_worked_example.py` asserts the math invariants (efficiency, sum-equality across variants, zero-on-unshocked for explicit-only) on synthetic data; `docs/methodology.md` carries no printed magnitudes to test (the worked example is qualitative). Specific Conditional Shapley magnitudes drift across `shap` library versions — do not reintroduce magnitude pinning without also pinning the shap version.
- **(Phase 18) The ridge is STANDARDIZED (unit-variance) with a raw-unit rescale.** `_solve_standardized_ridge` scales centered factor columns to unit variance, solves, then rescales `beta_raw = beta_std / sigma_f` — the rescale is LOAD-BEARING (it keeps `betas @ raw_decimal_shock` dimensionally unchanged; a column-scale-invariance test pins it). `alpha=0.1` now shrinks all directions homogeneously (`λ/(λ+0.1)` with standardized Gram diag ≈ T−1) instead of crushing low-vol factors. Zero-variance columns get exact-0.0 betas.
- **(Phase 18) Per-ticker NaN masks + 40-week floor.** Factor-side NaN rows drop globally; each ticker then regresses on its own valid rows (grouped by identical mask pattern — the all-complete case stays one solve). A ticker with `< MIN_REGRESSION_WEEKS = 40` valid weeks raises `InsufficientHistoryError` (a `RuntimeError` subclass) naming it → API maps to **422** at `/run`, `/adjust-shocks` (MUST precede the generic `RuntimeError → 422 rerun_required` clause there — mislabeling it would offer a rerun CTA that cannot help), and non-stream `/decompose`.
- **(Phase 18) `estimate_betas_for_portfolio` returns `(betas, stats)`.** `stats` is `dict[str, TickerRegressionStats]` (r2 in [0,1], n_obs, idio_vol_weekly — NOT annualized). `_estimate_betas_cash_aware` passes the tuple through with NO stats entry for CASH; `_benchmark_overlay` discards stats. `ScenarioResult.regression_quality` IS cached with the canonical (deterministic from the keyed vintage — unlike the never-cached NAV/benchmark overlays) and is recomputed on adjustments. `estimate_betas` remains a betas-only thin wrapper for tests.
- **(Phase 20) `ScenarioResult.analog_replay` is cached-with-canonical, factor-only, and preserved untouched by `adjust_scenario_shocks`.** Computed in `run_scenario` right after betas (`_analog_replay_block` → `app/factors/shocks.py::analog_replay_pnl`); deterministic from already-keyed inputs (betas ← `regression_spec`, event rows ← `events_version`), zero LLM involvement, additive-with-default → NO `PROMPT_VERSION` bump (Phase-13 precedent). Older cached/saved payloads deserialize with `None` — the UI renders "not computed", NEVER zero. NaN event returns (ETF predates the window) contribute exactly 0 and are excluded from `n_factors_covered`; event-return keys missing from the beta columns are ignored (vintage-subset betas in the replay harness).
- **(Phase 20) The engine-replay validation harness lives in `app/factors/engine_replay.py` and must stay LLM-free** (imports only `app.data.*` / `app.factors.*` / `app.config` — adding an `app.llm` import breaks its no-LLM guarantee). Regenerate BOTH artifacts (`scripts/engine_replay_output.json` + `docs/engine-replay-validation.md`, full regeneration — never hand-edit) with `uv run python scripts/run_engine_replay.py` whenever `REGRESSION_ESTIMATOR_ID`/ridge params, `app/factors/universe.py`, `data/historical_events.yaml`, or `sample_portfolio_weights.json` change; the artifact embeds `regression_spec`/`events_version`/factor-universe version/weights as-of so staleness is diff-visible. Vintage policy: factor columns with <40 non-NaN weeks at the vintage are DROPPED (never zero-filled — same rationale as the SHAP background) and disclosed per pair against the full 26-factor universe; pairs with post-vintage IPO holdings SKIP with a named reason rather than renormalizing around the hole. The network test in `tests/test_engine_replay.py` is `RUN_NETWORK_TESTS=1`-gated with semantic assertions only; sign asserts deliberately exclude q4-trade-war-2018 × defensive_mix (defensives were ~flat that quarter).
- **(Phase 21) `TickerRegressionStats` gained `r2_adj` / `p_eff` / `beta_se` — all defaulted None** so 3-field constructors (test mocks) stay valid. `r2_adj` (dof-honest, can be NEGATIVE) and `p_eff` (ridge effective dof `Σ s/(s+α)`) ARE persisted in the cached `TickerRegressionQuality`; `beta_se` (raw-unit ridge-sandwich SEs on `n−p_eff−1` dof, pinned against statsmodels OLS at α→0) is deliberately in-process only — do not add it to the cached schema without a rendering surface.
- **(Phase 21) `RIDGE_ALPHA` stays 0.1 — dated decision record, do NOT raise it without re-running the sweep.** `α ∈ {0.1,2,8,32}` over the engine-replay harness (2026-07-02, 48 pairs): MAE 3.97/3.90/4.26/5.03%, bias −2.13→−2.79% — flat to α=2, monotonic degradation beyond, because this ridge shrinks toward ZERO and understates betas. Under-determination is handled by disclosure (`r2_adj`/`p_eff` + `low_regression_dof` diagnostic), not blanket shrinkage; informative shrinkage targets (market/sector priors) are the future-work lever. Changing `α` self-invalidates caches via `regression_spec` AND lets in-TTL adjustments of pre-change canonicals recompute under the new value (mixed but TTL-bounded — accepted).
- **(Phase 21) `ScenarioResult.pnl_uncertainty` is the ±1σ idio band — dispersion FLOOR, never a confidence interval.** `band = √(Σ(wᵢσᵢ,idio)²)·√h`, `h` = median selected-analog window in weeks (`shocks.py::portfolio_idio_band` + `scenario.py::_pnl_uncertainty_block`). Shock-independent → cached with the canonical and recomputed to the IDENTICAL value on adjustments (horizon comes from `canonical.analog_event_returns`; old canonicals without it get None). Cross-name and cross-week independence assumptions bias it DOWN — keep the "floor, not CI" framing in any copy. Additive default field → NO PROMPT_VERSION bump.
- **(Phase 21) Three additive diagnostic kinds: `band_coverage`, `scenario_vs_replay`, `low_regression_dof`.** `generate_risk_diagnostics` gained an `analog_replay=` kwarg (canonical's block on the adjust path). `band_coverage` counts MATERIAL shocks only (|shock| ≥ 0.01) and warns when >½ are unbanded, else info. `scenario_vs_replay` is threshold-free (bounds = the analogs replayed on the book) with a 50bps dead zone (`SCENARIO_VS_REPLAY_TOLERANCE`). `low_regression_dof` fires below 3 observations per effective parameter and is silent when `p_eff` is absent (old payloads). The frontend `RiskDiagnostics` component renders kinds generically — extending the Literal + types.ts union is the whole UI wiring.
- **(Phase 22) The factor universe is 26 factors** — Phase 22 added `EFA` (market group) and `HYG`/`GLD`/`SHY` (macro group, decimal PRICE returns unlike the index-level TNX/DXY/VIX/OIL). Adding/removing factors changes `factor_universe_version` (cache lever) AND the descriptions feed the prompt payload → PROMPT_VERSION bump required (v10 here). New-factor launch dates (HYG 2007-04, GLD 2004-11, SHY 2002-07, EFA 2001-08) mean pre-launch analog events carry NaN for them — envelope `count` discloses it; never zero-fill. Frontend `FALLBACK_FACTORS` in `frontend/src/factors.ts` must be extended in the same change.
- **(Phase 22) The event registry is 31 events with 12 valid tags** (`technology`, `volatility`, `credit`, `disaster` added). The dot-com and 9/11 events contribute ENVELOPE evidence only — every sample book skips them in the replay harness (holdings didn't exist / <40 weeks of pre-2000 history), which is expected and disclosed per pair. `events_version` (YAML content hash) invalidates caches on any registry edit.
- **(Phase 22) Dimson lag was evaluated and REJECTED by harness evidence — do not add it without new evidence.** The attenuation hypothesis predicts Japan-book understatement; the harness showed +2.2% OVERstatement (US books −2.4% to −4.3%). Adding `EFA` instead cut Japan bias to +0.7% and MAE 5.9%→5.1%. The dated decision record lives in `docs/methodology.md` (Japan currency paragraph).
- **(Phase 23) Stressed betas were evaluated and REJECTED by harness evidence — do not add downside-weighted (WLS) or informative-target shrinkage without re-running the experiment.** Downside-WLS (bottom-decile SPY stress weeks upweighted k×) degrades MAE/bias/r MONOTONICALLY in k (MAE 3.25%→4.18%, r 0.976→0.955 by k=25) and trades a small Japan-bias gain for degradation everywhere else; the informative-target ridge (β=1 prior on SPY/EFA) is a noise-level wash at α=2 and worse beyond — it does NOT rescue heavier shrinkage. Analog-window estimation is the k→∞ WLS limit (monotonically harmful gradient), so it was not separately implemented. Estimator stays `ridge-std-v2|alpha=0.1`; `REGRESSION_ESTIMATOR_ID` unbumped. Dated decision record in `docs/methodology.md` (beta-estimation section).
- **(Phase 23) Back-to-back full engine-replay harness runs WILL rate-limit Yahoo.** Deep-vintage factor windows contain pre-launch ETFs → the batch comes back incomplete → the Phase-19 "complete batches only" market-cache rule never caches those windows → EVERY full harness run re-fires ~20+ multi-ticker live downloads no matter how warm the GCS cache is. One run at a time is fine; repeated sweeps (estimator experiments) must memoize the four fetch functions in-process for the sweep's lifetime, or space runs out. A rate-limited window mid-run also poisons that pipeline's factor history — which is why the warm cache must never memoize degraded results (see the warm-cache bullet).
- **(Phase 25) The full events × factors matrix is process-cached in `warm_cache.get_event_returns_matrix()`, keyed on `events_version`.** Historical windows never change, and deep-vintage batches are UNCACHEABLE in the GCS market cache (pre-launch ETFs → incomplete batches → never written), so an uncached full-registry fetch re-fires ~20+ live Yahoo downloads per request. Degraded-result rule matches the factor cache: a row that is ENTIRELY NaN means the provider returned nothing for that event (transient) — returned but never memoized; legitimate pre-launch NaN holes are partial rows and cache fine. A registry edit invalidates via the version key.
- **(Phase 26) Ladder rung construction is per-factor adverse/favorable edges, NOT all-p10/all-p90.** P&L is linear in each shock so the min/max over the `[p10,p90]^K` box is at a vertex chosen by the SIGN of this book's exposure `E_f = Σwᵢβᵢ,f`; an all-p10/all-p90 pair does not bracket the base when any exposure is negative. Banded = envelope `count >= MIN_ENVELOPE_COUNT_FOR_BAND_CHECK` (=3, imported from `app/llm/validation.py` — the SAME gate as `validate_factor_overrides`/the sliders); low-evidence shocks held at proposed values (`n_held`), removed (0.0) shocks contribute nothing, periphery total rides along. `worst ≤ base ≤ best` and `base_pnl == portfolio_pnl.total_pnl` are pinned invariants (`tests/test_scenario.py`, `tests/test_adjust_manual.py`).
- **(Phase 30) The warm-cache getters are SINGLE-FLIGHT** — `get_factor_returns_with_history` and `get_event_returns_matrix` hold a dedicated fetch lock across the WHOLE fan-out (double-checked against the result dict), so a request racing the background warm (or another request) WAITS and reuses the in-flight result. Without it, concurrent duplicate fan-outs contend on Yahoo rate limits and made the racing request SLOWER than no warm at all (observed live: 150s events-replay right after deploy). A degraded fetch releases the lock and the next caller retries serially — the never-memoize-degraded rule is unchanged. Pinned in `tests/test_warm_cache.py`.
