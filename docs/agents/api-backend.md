# nami — FastAPI backend & persistence (agent notes)

Owns `app/api/*` and `app/observability/*`: endpoints, SSE stream contracts,
auth/access, rate limiting + budget metering, the coded error contract, and
Firestore persistence. Change a contract here → update this doc in the same
commit.

## Gotchas

- **CSV/custom portfolio rules** live in `app/api/portfolio_validation.py`: uppercase tickers (`.T` suffixes preserved); reject blanks / duplicates / negatives / non-finite; totals near 1.0 accepted as decimals OR near 100 as percentages (auto-normalized to decimals). Anything else is rejected.
- **Admin access is an HTTP-only signed cookie**, not frontend-only state. `app/api/security.py` signs the admin cookie with the configured passcode; changing the passcode invalidates existing admin cookies.
- **`run_scenario` accepts an optional `progress: Callable[[str, str], None]` callback** (Phase 10). It fires `progress(stage, status)` at stage boundaries: `cache_check`, `cache_hit`, `market`, `analogs`, `envelope`, `narrative`, `betas`, `attribution`. The SSE endpoint at `/api/scenarios/run-stream` wraps this — the worker thread runs `run_scenario(progress=...)` and a `queue.Queue` carries events to the FastAPI generator. `run_scenario` itself NEVER emits `"done"`; the SSE wrapper emits `{"stage":"done","result":...}` after the function returns.
- **`ScenarioRunResponse.cache_key` is the server's provenance handle.** The server computes it from `compute_scenario_cache_key(scenario_text, portfolio, market_date=...)` and returns it on both `/api/scenarios/run` and `/api/scenarios/run-stream`. The client echoes it back on `/api/scenarios/adjust-shocks`; the server re-fetches the canonical result and rejects with HTTP 410 if the cache TTL has expired. Don't trust client-supplied `prior_result` payloads — they're tamperable.
- **Firestore vs GCS roles in Phase 11.** GCS scenario_cache (existing) is the de-dup layer for identical-key runs (7-day TTL). Firestore (new, `saved_scenarios/{id}`, `portfolios/{id}/snapshots/{id}`) is user-facing persistence: tagged, named, permalinked, durable beyond the GCS TTL. Saved records store the FULL `ScenarioResult` + analog event details + reproducibility metadata INLINE. Do not rely on a Firestore record to dereference a GCS cache_key — the GCS entry may have expired.
- **Saved scenarios store everything inline (no FK to live state).** `portfolio_holdings` inline (sample portfolio defs in code can drift; saved analytics shouldn't). `analog_events_snapshot` inline (the events YAML can change). `reproducibility` inline (full block: model_id, prompt_version, factor_universe_version, events_version, beta_lookback_weeks, ridge_alpha, narrative_mode, both as-of dates, selected_event_ids, portfolio holdings + key, market data source, nami engine version). Rendering a saved scenario must NOT consult the live event registry or the GCS cache.
- **Firestore composite index requirement.** Tag-filtered scenario listing needs `tags array_contains + created_at DESC`. Missing-index errors are caught and re-raised as `RuntimeError` with the exact `gcloud firestore indexes composite create` command. See [app/data/firestore_store.py::FirestoreStore.list_scenarios](app/data/firestore_store.py).
- **Firestore 1MB doc limit hard-guarded at 900KB safety margin.** Saves over that raise `ValueError` → HTTP 413. Observed payloads << 100KB; the guard catches future regressions of fat ScenarioResult payloads.
- **Permalink `/saved/{id}` (URL: `?saved=<id>`) is admin-protected, NOT a public share link.** Incognito visitors see a 403. UI copy makes this explicit. Multi-user via Firebase Auth would change this in v2.
- **Holdings look-ahead guard for portfolio snapshots.** `/api/portfolios/{id}/snapshots` rejects future-dated snapshots with 422. A separate cross-validation (`portfolio_snapshot.as_of_date <= scenario_as_of_date`) should be wired when running a scenario against a saved snapshot — currently this happens implicitly when the UI populates holdings from a snapshot and the user picks an as-of date; an explicit server-side guard against `snapshot.as_of_date > scenario_as_of_date` would be a v1.1 hardening.
- **Portfolio snapshots are weight-only in v1.** Shares/MTM mode disables snapshotting (`PortfolioHistoryPanel` `snapshotDisabledReason`); recording share units in snapshots is a v1.1 follow-up.
- **(Phase 16) The paid-Gemini budget is enforced at ONE chokepoint: `GeminiClient._generate_content`.** All 6 `generate_content` call sites route through it; `app/observability/metering.py::MeteredGeminiClient` overrides it to **reserve → call → reconcile** against a Firestore-transactional `usage_daily/{YYYY-MM-DD}` doc. The endpoints build the metered client via `_metered_gemini()` and inject it through `run_scenario`/`adjust_scenario_shocks`/`compute_narrative_shapley`'s existing `gemini=` param — so internal fan-out (retries, 2^N decomposition reruns) and cache-miss-only spend are all counted, while a cache hit (zero calls) costs nothing. Do NOT wrap the public methods instead — they'd miss calls buried inside `propose_shocks_with_retry`. A real cap hit raises `BudgetExceededError` (→ 429, NOT a RuntimeError/ValueError so the existing handlers don't swallow it); a store/infra error fails OPEN (the rate limiter + run cap are backstops). Reserve is the atomic gate against concurrent overspend; failed calls reconcile conservatively (book the estimate) so they can't escape the cap.
- **(Phase 16) Rate limiting (`slowapi`) is decorator-only on the 5 paid endpoints + `/api/auth/unlock`.** Never throttle the `/{path:path}` SPA catch-all or `/assets` (the app must always load). State is per-instance in memory (fine for max-2 instances). The trusted client IP comes from `app/api/middleware.py::client_ip` (reads `X-Forwarded-For` RIGHT-to-LEFT, returns first publicly-routable hop) — the SAME function keys the durable lockout, so a spoofed left-most XFF entry can't reset either counter. Tests disable the limiter via the autouse `_disable_rate_limit` fixture in `tests/conftest.py`; the dedicated rate-limit test re-enables it with a unique source IP.
- **(Phase 16) `/api/health` is liveness (always 200, no I/O); `/api/ready` is the dependency probe** (bounded `ThreadPoolExecutor` + `future.result(timeout=5)` over Firestore/GCS/Gemini, config+ADC only — NO paid generation). Keep them separate: Cloud Run health checks must stay fast/dependency-free.
- **(Phase 16) `request_id`/`ip_hash` live in `contextvars` (`app/observability/context.py`).** The HTTP middleware sets them and existing module loggers pick them up with no signature change. SSE worker threads do NOT inherit contextvars — both stream endpoints capture `ctx = contextvars.copy_context()` and run the worker via `ctx.run(worker)`; the same propagation carries the metered client. Never store raw IPs — only `hash_ip(...)`.
- **(Phase 16) New Firestore collections: `usage_daily`, `auth_throttle`, `audit_log`.** All single-doc get/set/transaction or single-field order-by — NO composite index needed. `purge_all()` walks the `snapshots` subcollections (chunked under the 500-op batch limit) and deliberately does NOT delete `audit_log` (the trail survives a purge; a final `admin.purge` event is written after). The `InMemoryFirestoreStore` mirrors every method (lock-protected) so the budget/lockout/audit tests run without real Firestore — endpoint tests inject it via `monkeypatch.setattr("app.api.main._firestore_store", ...)` (now also in `test_api.py`/`test_api_adjust.py` because the run endpoints touch the store for the run cap).
- **(Phase 16) `/api/status` REUSES `DISCLAIMER_SHORT`** — `est_cost_today_usd` is admin-gated (None for visitors); `/api/usage` is admin-only. The UX additions (Impact-summary readout, chips, ⌘K palette) are additive: the palette is an accelerator whose commands all have visible controls. No `PROMPT_VERSION` bump for Phase 16.
- **(Phase 19) `X-Error-Code` header + the SSE `code` field are the machine-readable error contract; detail strings are display-only.** Never reintroduce client-side detail-string matching — the adjust-shocks 422 detail is LLM free text (`rejection_reason`). Header codes (emitted only where status alone is ambiguous): `rate_limited`, `budget_exhausted`, `run_cap`, `rerun_required`, `marking_unavailable`. SSE error-event codes: `run_cap`, `budget_exhausted`, `marking_unavailable`, `validation`; an absent code maps to client kind `unknown`, NEVER `network` (an in-band SSE error arrived over a healthy HTTP 200 connection). `app/api/errors.py::http_error` is the coded-raise helper; `frontend/src/api.ts::deriveErrorKind` is the client map; `frontend/src/errorCopy.ts` is the copy layer.
- **(Phase 19) Paid endpoints must keep their FULL pipeline call inside the `except BudgetExceededError` try** — the metered client raises mid-pipeline (e.g. during decomposition's 2^N subset calls), not just at `_metered_gemini()` construction. The decompose endpoint had exactly this hole (mid-compute budget hit → 500); pinned by `tests/test_api_errors.py::test_decompose_budget_exceeded_mid_compute_is_429`.
- **(Phase 19) `RunCapExceededError` MUST subclass `BudgetExceededError`** (`app/observability/metering.py`) — the endpoints' existing `except BudgetExceededError` handlers catch it; the subclass exists only so `_budget_http_error` can emit `run_cap` vs `budget_exhausted`.
- **(Phase 19) Saved-analytics endpoints wrap store access in `_store_errors()`** (app/api/main.py) — Firestore failures are google-cloud exception types, NOT RuntimeError (missing database, permissions, bad local creds), and previously leaked as bare 500 "Internal Server Error". The context manager maps them to a coded 503 `unavailable` whose detail names the real cause (incl. the store's composite-index gcloud instructions); HTTPExceptions (404/413/422) pass through untouched. `/api/status` always returns 200 and now honestly flips `ready=false` when the store is unreachable (the old suppress never flipped it). Frontend kind `unavailable` passes meaningful detail through (generic copy only for bare status lines).
- **(Phase 24) `/api/portfolios/profile` is free (zero-Gemini) but rate-limited.** `compute_book_profile` lives in `app/llm/scenario.py` (it reuses `_estimate_betas_cash_aware` + the warm factor cache — LLM-free despite the module), is deliberately NOT metered, and takes the `llm_limit` throttle only because it fans out to yfinance. Visitors: sample books only (custom holdings → 403); admins: custom WEIGHT books (shares/MTM books cannot be profiled — their weights are price-derived server-side only during a run; the UI disables the CTA with a caption). Error mapping mirrors `/run`: `MarkingError` → 503 `marking_unavailable`, `InsufficientHistoryError` → 422, bare `RuntimeError` → coded 503 `unavailable` (clause stays LAST).
- **(Phase 25) `/api/portfolios/events-replay` mirrors the profile endpoint — new free engine endpoints must reuse `_run_free_engine_endpoint`** (app/api/main.py: visitor field guard + `_resolve_portfolio` + the /run-mirroring error ladder in ONE place). `compute_events_replay` pushes every registry event's realized factor row through the CURRENT book's betas via `analog_replay_pnl` (current betas ≠ the vintage betas the validation harness uses — different questions), sorts worst-first server-side, and shares `_book_betas` with `compute_book_profile`.
- **(Phase 26) `AccessResponse.sample_weights_as_of`** (from `sample_as_of()`) backs the backdated-banner PIT-drift line; the line renders only for sample books (`portfolio_key !== "custom"` — custom holdings are user-supplied vintage). Frontend `AccessResponse.sample_weights_as_of` is OPTIONAL in types.ts so existing test fixtures stay valid (the Phase-15 fixture-churn lesson).
- **(Phase 29) `SseAwareGZipMiddleware` (app/api/main.py) compresses everything EXCEPT the two SSE stream paths.** Gzip's deflate buffer holds small frames until a block fills — compressing `run-stream`/`decompose-stream` would freeze the progress stepper mid-run. Cloud Run does NOT compress for you; this middleware is the only compression layer. If you add a new streaming endpoint, add its path to `_SSE_STREAM_PATHS`.
- **(Phase 29) Cache-Control is split by content-addressing:** `/assets/*` (hashed filenames) serve `public, max-age=31536000, immutable` via `ImmutableStaticFiles`; `index.html`/SPA catch-all serve `no-cache` (revalidate every visit — deploys must not strand old clients); root public files (favicon/manifest/robots, un-hashed) get `max-age=3600`. Don't "optimize" the index to a positive max-age. NOT a bug: on the deployed service the asset header reads `private, max-age=31536000, immutable` — Cloud Run's session-affinity layer (the GAESA cookie) rewrites `public` → `private` in transit; browser caching (the goal — there is no shared CDN) is unaffected, so don't chase the diff.
- **(Phase 29) Unknown `/api/*` paths return a JSON 404 from the catch-all** instead of the masked HTML-200 SPA shell. The catch-all is registered with `methods=["GET", "HEAD"]` — FastAPI does NOT auto-derive HEAD from GET, and uptime monitors probe `/` with HEAD (a 405 there reads as an outage).
- **(Phase 30) SSE streams MUST heartbeat — `_sse_stream_response` (app/api/main.py) is the shared factory for both stream endpoints.** It drains the worker queue with `events_q.get(timeout=_SSE_HEARTBEAT_SECONDS)` (10s) and yields an SSE comment frame `": keepalive\n\n"` on `queue.Empty`; the StreamingResponse carries `Cache-Control: no-cache` + `X-Accel-Buffering: no`. Reason: the run stream's `narrative` stage is a 30-90s zero-byte gap (2-3 sequential Gemini calls) and middleboxes/proxies sever silent connections — the live symptom was `kind: "network"` errors while the server finished with a 200. A NEW streaming endpoint must use this factory AND join `_SSE_STREAM_PATHS` (the gzip bypass).
- **(Phase 30, Phase 35) Startup warming is a BACKGROUND daemon thread** (`_background_warm` in main.py: `warm_cache.warm()` then `warm_cache.get_event_returns_matrix()`, each suppressed). Lifespan must NOT block on network fetches. The complete event matrix now persists in GCS for 30 days, so a new instance normally loads parquet instead of repeating the ~2-minute Yahoo fan-out; the first population after expiry remains cold and background-only.
- **(Quant V2) Run requests carry `horizon_days` and `severity` with defaults 21 and 1.0.** Accepted values are 5/21/63 and 1.0/1.5/2.0. The fields are semantic only in `quant_v2`; legacy keeps its old cache/math contract. `/api/access` and `/api/status` expose the configured `engine_mode`, and every result records its actual `engine_mode`, `engine_version`, horizon, severity, support diagnostics, public-data provenance, and methodology range when applicable.
- **(Quant V2) Unsupported legacy edit surfaces fail at the API boundary.** Shock adjustment and narrative decomposition return 422 for Quant results instead of attempting to hydrate incompatible Shapley/periphery state. Saved legacy payloads remain accepted because the added result fields are optional/defaulted. Reproducibility may report `French Data Library + FRED + yfinance`; keep the backend schema and frontend union synchronized.
