## Gotchas

- **(Phase 31i) Attribution surface: the Drivers waterfall is LOCKED to `selectMainAttribution` (explicit conditional Shapley, naive fallback with an on-screen degraded caption)** with a By factor / By group zoom (same map, client rollup). App no longer owns an `attributionMethod` state — the readout computes its own preferred method, ComparisonPanel computes `commonAttributionMethod` internally, and the audit radiogroup lives locally in `results/MethodologyDiagnostics.tsx` (Advanced tab, above theme sensitivity; keeps the correlation-credit caveat verbatim, test-pinned). Don't reintroduce a method toggle on the main surface.
- **(Phase 31i, Phase 33) Chart fullscreen = `useFullscreen(ref, { surface })` with TWO presentation modes on the SAME node.** Where the native Fullscreen API is available it drives it (browser owns Esc/backdrop, so the two-overlays-Esc rule is untouched). Where it is absent (`document.fullscreenEnabled` falsy — iPhone Safari, jsdom) OR `requestFullscreen()` rejects at runtime, `toggle()` enters an app-owned **expanded-card modal** on the same node instead of no-opping — `isFullscreen` is true in BOTH modes and callers must not care which. Fallback contract (mirrors `useOverlay` scroll-lock + `OverlayShell` focus capture/restore): the node gets `is-card-expanded` (keeps its static `fullscreen-surface` class) + `role="dialog"`/`aria-modal`/`aria-label={surface}`, `document.documentElement` gets `has-expanded-card`, body scroll locks, a window Esc listener collapses it, `useFocusTrap(ref, active)` contains Tab, and the opener's focus is restored on exit; ALL of that is removed on collapse AND on unmount. **Singleton**: a module-level registry keeps at most one expanded card app-wide (expanding a second collapses the first); `closeExpandedCard()` is exported and JOINS the overlay mutual exclusion — `useOverlayManager`'s `openMethodology`/`openRailDrawer`/`openOpsDrawer`/`requestPurge`/⌘K all call it, and `openCommandPalette()` is now the ONLY palette opener (direct `commandPalette.open` calls are forbidden — they'd skip the card-collapse). Every overlay opener actually calls `dismissFullscreenSurfaces()` (`state/useOverlayManager.ts`), which closes the fallback expanded card AND exits native fullscreen — without it, the palette (and other overlays) could open invisibly underneath a still-fullscreened native element. `.fullscreen-surface:fullscreen`/`::backdrop` style the native mode; `.fullscreen-surface.is-card-expanded` is `position: fixed` at `--z-overlay` (below the toast tier). A transition dispatches a window resize next frame (BOTH entry and exit, incl. cleanup-on-unmount) so Plotly re-fits — native mode only fires on the owning element's OWN transition (`fullscreenchange` hits `document` for every mounted hook instance). Headless-test note: synthetic Esc does NOT exit *native* fullscreen — exit via the button in Playwright; the *fallback* IS jsdom-drivable (see `useFullscreen.test.ts`). The affordance is the shared `FullscreenButton` (`frontend/src/FullscreenButton.tsx`), now universal (no longer hides on `!supported`) — `surface`-named "Expand {surface}"/"Collapse {surface}" labels. Chart height is the pure `fullscreenChartHeight(isFullscreen, base, viewportHeight?)` helper plus `useViewportHeight(active)`, both in `useFullscreen.ts`. **(Phase 33, Task 3) The affordance fans out past the two chart cards.** `CollapsibleCard` gains `fullscreenable?: boolean` + `surface?: string` (surface required whenever `fullscreenable` is set — it labels the button and the fallback's `aria-label`); the hook is called UNCONDITIONALLY (never gated by the prop) so hook order never depends on it. Opted-in table cards: `results/ResultsPanel.tsx`'s factor-shocks, name-level-contribution, position-valuation, narrative/analogs, and theme-sensitivity cards, plus `results/ExposureBreakdown.tsx`. Three non-collapsible surfaces wire the hook + `FullscreenButton` directly on their root section: `KnowYourBook.tsx` (one hook for the whole card, button beside the profile/events segmented control), `ComparisonPanel.tsx` (button beside Unpin), and `EvidenceBlock.tsx` (button beside the eyebrow in a new `.evidence-head` row — its hooks sit ABOVE the `if (!gauge) return null` early return, since the gauge is data-dependent and a hook must never be gated by a value derived from props). **Anti-trap contract on `CollapsibleCard` (blocking — an expanded card in fallback mode is a touch user's ONLY exit, no hardware Esc exists):** the action cluster (and the fullscreen button inside it) renders whenever `(open && action) || (fullscreenable && (open || fs.isFullscreen))` — so it never disappears while `fs.isFullscreen` is true, regardless of `open`. AND the collapse header's click handler checks `fullscreenable && fs.isFullscreen` FIRST: while fullscreened it calls `fs.toggle()` (exits fullscreen) instead of `setOpen(false)` — collapsing the body out from under the only exit control would strand the user. Test-pinned in `CollapsibleCard.test.tsx` (verified RED without the guard: the expanded-card class survives a header click that should have exited it). **Deliberately excluded**: the OpsDrawer audit log (already inside a `useOverlay` drawer — nesting would create two window-level Esc owners), the answer-band readout, and `RunProgress`. `EvidenceBlock` (and everything else in the answer band) must stay `CollapsibleCard`-free.
- **(Phase 34) Programmatic navigation exits native fullscreen only when its owner will be hidden.** `exitNativeFullscreenIfOwnerWillHide()` receives affected mounted-tab scopes plus their next-visible panels; App uses it at run-completion, saved-open, and completion-toast navigation seams. A non-Drivers result card exits before the Drivers snap, while a natively-fullscreen Drivers waterfall remains fullscreen. Fullscreen outside an unrelated changing scope (for example Book analytics while only the results sub-tab changes) is untouched. The app-owned fallback still collapses eagerly with `closeExpandedCard()` because its body lock and window-Esc owner cannot safely survive a hidden panel.

- **(Phase 33, Task 6) The results toolbar carries two quick actions after the Pin control: "Adjust shocks" and "Edit & re-run".** *Adjust shocks* renders on the SAME gate as the Adjust tab (`showAdjust && safeTab !== "adjust"`) and just `changeTab("adjust")` — it can never point at a missing tab. *Edit & re-run* renders whenever App passes `onEditRerun` and rebuilds the composer from that result via the pure `frontend/src/rerunDraft.ts::buildRerunDraft(result, { sampleKeys, isAdmin })` (no React imports — App supplies context, the helper decides, `handleEditRerun` applies + navigates + focuses). **This is the saved-scenario iteration path**: saved/permalinked results reopen without a `cache_key`, so they have no Adjust tab; Edit & re-run is deliberately their ONLY iteration path. **We do NOT resurrect `cache_key`s** — the GCS scenario cache has a 7-day TTL, so an adjust-on-saved would be a 410 trap; full-state re-run is durable instead. Hydration rules the helper is exhaustively unit-tested on (`rerunDraft.test.ts`): a recognized sample key restores as a sample selection, anything else (incl. `"custom"`) rebuilds a **weights-mode** book from `portfolio_holdings` (shares/MTM results convert to their server-derived weights — the share counts are never resurrected; `sharesConversion` triggers a disclosure toast and `nav` rides along); `benchmark_ticker` rides along; **backdated is signalled by `narrative_mode === "analog_only"`, NOT a date comparison** (a reopened live run keeps its own day's close as `requested_as_of_date`, now in the past — a date compare would mis-flag every reopened live scenario as backdated), and a live restore CLEARS the as-of picker back to the live anchor so a stale pick can't silently backdate the re-run. `needsAdmin` (custom OR backdated, checked against the session's `isAdmin`) makes `handleEditRerun` show an unlock toast + `focusUnlock()` and STOP — never a silent partial hydration. `handleEditRerun` MUST NOT touch `resultEnvelope` / `canonicalSnapshot` / `pinnedEnvelope` (the pin invariant below). The toasts here are user-initiated feedback, so they are regular (announced), NOT `silent`.
- **(Phase 33, Task 6) The AdjustmentPanel "rerun required" / expired prefill routes through `handleEditRerun`.** `onPrefillRerun(text)` now calls `handleEditRerun(canonicalSnapshot ?? resultEnvelope.result, text)` with a `textOverride`, so the combined original+adjustment text still prefills the composer but the book / benchmark / as-of context rides along too (the old handler only set scenario text). Keep the `textOverride` param — it is the single path shared by the toolbar button and the adjust-panel prefill.

"Canonical -> Current" renders only on changed rows. `rows` still carries EVERY canonical factor — the server rejects missing/extra keys.
- **(Phase 31g) The live adjust preview (`results/adjustPreview.ts`) is EXACT for the headline** because the engine total IS the naive sum: `total = periphery + Σ by_factor_naive[f]`, so `exposure_f = naive_f / shock_f` and edits scale linearly. Verified against a real server recalc (preview 14.22% == recalculated 14.22%). It reads the CURRENT displayed result (not the canonical) so exposures match what's on screen; returns null when a factor is re-tuned FROM zero (0/0 exposure); attribution maps still need Recalculate — the strip says so.
- **(Phase 31g) `.adjust-tunable` uses `repeat(auto-fill, minmax(min(360px, 100%), 1fr))`** — the `min(360px, 100%)` guard keeps the 360px track floor from forcing horizontal overflow on phones (same min-content trap as the tables). `kr-chip` gets a 44px touch target ≤1079.98px.
- **Results-tab labels must keep `shock applied` vs `contrib to P&L` distinct.** "Shock applied" is the magnitude the LLM proposed for a factor; "contrib to P&L" is `(Σᵢ wᵢ · βᵢ,f) · shock[f]` — the weighted contribution to the total. Never collapse these into a single number.
- **SSE client uses `fetch` + `ReadableStream`, not `EventSource`.** `frontend/src/api.ts::runScenarioStream` reads chunks and splits on `\n\n` frames manually. Reason: `EventSource` is GET-only and doesn't carry POST bodies; the chunked-reader approach also forwards cookies cleanly on same-origin requests. The terminal event is `{"stage":"done","result":...}`; the stream also handles `{"stage":"error","message":...}` by rejecting the promise.
- **Methodology UI is a slide-in drawer, not an inline panel.** `frontend/src/MethodologyDrawer.tsx` renders a right-side drawer over a backdrop with body-scroll-lock + Esc-to-close (`useMethodologyDrawer.ts`). It auto-parses `docs/methodology.md` by splitting on `\n---\n`, slug-ifies section titles, renders an accordion (one expanded section at a time by default, multiple allowed). Deep links from anywhere in the app open the drawer at a specific section: `methodologyDrawer.open("factor-attribution")` matches the first section whose `slug.includes("factor-attribution")`. **The drawer parses markdown sections at render time, so adding a new section to `docs/methodology.md` is enough — no code change needed for the drawer to pick it up.** New deep-link targets DO need a matching `methodologyDrawer.open("<slug>")` call site.

The `AttributionGuide` component under the attribution toggle deep-links to `factor-attribution`. Both pass through `ResultsPanel`'s new `onOpenMethodology: (section?: string) => void` prop.

- **`docs/methodology.md` has a "Worked example" heading** inside the Factor-attribution section that the drawer's `parseMethodologySections` flags (via the `hasExample` regex `/worked example/i`) and the CSS highlights with a subtle border accent. It is now a short **qualitative** 2-factor table (the old numeric 3×2 walk-through was deliberately removed — printed magnitudes were a drift magnet; the invariants are pinned synthetically in `tests/test_methodology_worked_example.py`). Keep the heading text when editing, keep the `factor-attribution` section self-contained for the AttributionGuide deep-link, and do NOT reintroduce a numeric example without pinning the shap version.
- **The rail (`<aside className="rail">` with AccessPanel + PortfolioPanel) is conditionally rendered.** `frontend/src/App.tsx` uses `useMediaQuery("(max-width: 1079.98px)")` to choose between inline `<aside>` (desktop) and `<RailDrawer>` (tablet/phone, opened via hamburger). The shared JSX is extracted into a `const railContent` and passed as `children` to RailDrawer. Mounting only one prevents duplicated component instances with diverging local state. The two drawers (rail + methodology) are mutually exclusive — opening one closes the other via `openMethodology` / `openRailDrawer` helpers in App.tsx. **The compact drawer self-closes after a COMPLETED choice** — picking a sample book (`setPortfolioKey`) or a successful admin unlock (an `onAccessChange` call with `intentional: true` landing on `access_mode === "admin"`) — via `wrapWithDrawerClose`/`isCompletedUnlock` in `frontend/src/railDrawerClose.ts`, applied ONLY at the App boundary (RailContent/PortfolioPanel/AccessPanel stay unaware). It deliberately does NOT close on custom-book edits, mode switches, CSV upload, or lock — those leave more work to do inside the drawer, and a lock isn't a completed pick. Desktop's drawer is never open, so the wrappers are no-ops there. The command palette's "Open portfolio & access setup" command (`focusSetup`) is compact-only for opening the drawer — on desktop, where the inline rail is already mounted, opening the drawer would double-mount AccessPanel/PortfolioPanel (the "forked state" trap), so it instead scrolls the inline rail's sample-book `<select>` into view and focuses it, mirroring `focusUnlock`'s two-layout branch.
- **`useOverlay` is the shared overlay primitive** ([frontend/src/useOverlay.ts](frontend/src/useOverlay.ts)). It handles `isOpen` state, body-scroll-lock, and Esc-to-close. Critically, the `onClose` callback fires BEFORE `isOpen` flips to false — so consumer-specific cleanup (like clearing `initialSection` on the methodology drawer) runs deterministically for both the explicit `close()` call AND the Esc-key path. `useMethodologyDrawer` is now a thin wrapper around `useOverlay({ onClose: () => setInitialSection(null) })`.
- **`useMediaQuery` is SSR-safe** ([frontend/src/useMediaQuery.ts](frontend/src/useMediaQuery.ts)). Returns `false` during SSR; subscribes to `change` events client-side. Used for the rail-render switch in App.tsx and for the Plotly mobile-layout switch in `ResultsPanel`.

- **Overlay state and overlay shell are separate.** Parent hooks (`useOverlay()`) own body-scroll-lock + Esc-to-close. `OverlayShell` owns backdrop click, `role="dialog" aria-modal="true"`, focus trapping, initial focus, focus return, and the optional close header. - **`useFocusTrap(ref, active)` ([frontend/src/useFocusTrap.ts](frontend/src/useFocusTrap.ts)) adds ONLY Tab-cycling containment.** It composes with `useOverlay` (scroll-lock + Esc) through `OverlayShell`; do not duplicate it inside individual drawers/dialogs. - **AdjustmentPanel slider range is exactly `[p10, p90]`; the Remove button is the only path to 0.** The valid domain `[p10,p90] ∪ {0}` is disjoint when the envelope is one-signed, which a single continuous slider can't represent.
